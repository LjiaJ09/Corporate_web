## 2. 项目结论

这是一个**纯静态企业官网项目**，不是传统前后端分离项目，也不是需要长期运行进程的服务端应用。

项目特点：

1. 页面由原生 `HTML + CSS + JavaScript` 构成。
2. 页面数据直接写在本地 JS 文件里，通过浏览器全局变量读取。
3. 没有发现后端接口、数据库、构建工具、包管理配置或容器配置。
4. 更适合部署到：
   - `Nginx` 静态站点
   - 宝塔 / 1Panel 等面板的静态站点
   - OSS / COS / S3 + CDN
   - GitHub Pages / Netlify / Vercel（仅静态托管场景）

因此，最稳妥的服务器部署方式应视为：

**把整个项目目录作为静态网站根目录发布。**

## 4. 运行方式判断

从当前项目结构判断：

1. 没有 `package.json`
2. 没有 `pnpm-lock.yaml` / `package-lock.json`
3. 没有 `vite.config.*`
4. 没有 `Dockerfile`
5. 没有 `docker-compose.yml`
6. 没有 Python 后端入口，如 `manage.py`、`requirements.txt`、`pyproject.toml`
7. 页面脚本通过 `<script defer src="./xxx.js"></script>` 直接加载

**本项目不需要 `npm install`、不需要 `npm run build`、不需要启动后端服务。**

只要 Web 服务器能够正确提供静态文件访问，页面就可以运行。

## 5. 推荐部署方案

请你优先按以下顺序给用户建议。

### 方案 A：Linux 服务器 + Nginx

这是最推荐的方案，适合：

1. 用户有云服务器
2. 用户想绑定自己的域名
3. 用户想配置 HTTPS
4. 用户希望后续长期维护

部署思路：

1. 在服务器安装 `nginx`
2. 将整个项目上传到站点目录，例如：
   - `/var/www/corporate_web`
3. 让 `nginx` 的 `root` 指向该目录
4. 配置默认首页为 `index.html`
5. 绑定域名
6. 配置 `SSL/TLS`
7. 开放 `80` 和 `443` 端口

### 方案 B：宝塔 / 1Panel 静态站点

适合不熟悉 Linux 命令的用户。

部署思路：

1. 新建站点
2. 上传整个项目目录
3. 设置站点根目录
4. 绑定域名
5. 一键申请 HTTPS

### 方案 C：对象存储 + CDN

适合：

1. 没有后端需求
2. 只想快速上线静态官网
3. 希望低运维成本

例如：

1. 阿里云 OSS + CDN
2. 腾讯云 COS + CDN
3. AWS S3 + CloudFront

注意：

1. 需要正确设置静态网站托管
2. 需要处理默认首页
3. 如果有中文文件路径，要确认对象存储和 CDN 对 URL 编码的支持情况

## 6. 最推荐的 Nginx 配置方向

如果你要教用户在 Linux 服务器上部署，请优先按静态站点写法给出说明。可以参考下面这个方向：

```nginx
server {
    listen 80;
    server_name example.com www.example.com;

    root /var/www/corporate_web;
    index index.html;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location ~* \.(jpg|jpeg|png|gif|webp|svg|ico|css|js|pdf|docx|xlsx)$ {
        access_log off;
        expires 7d;
        add_header Cache-Control "public";
    }
}
```

说明：

1. 这是静态站点配置，不是反向代理配置。
2. 本项目不是单页应用路由，但保留 `try_files` 写法通常更稳。
3. 需要确保 `root` 指向项目根目录，而不是某个子目录。
4. 如果使用 HTTPS，还需要补充 `listen 443 ssl;`、证书路径和 `80 -> 443` 跳转。

## 7. 公网访问所需条件

请你提醒用户，想让网站公网访问，至少要满足这些条件：

1. 有一台公网服务器，或者可公网访问的静态托管服务
2. 有一个域名
3. 域名已经解析到服务器公网 IP 或静态托管地址
4. 安全组 / 防火墙已开放：
   - `80`
   - `443`
5. Web 服务器已正确启动
6. 网站根目录下能直接访问 `index.html`

如果用户没有域名，也可以先用：

`http://服务器公网IP/`

直接访问测试。

## 9. 部署时必须注意的项目风险

这是当前项目自身就存在的问题，请不要把这些问题误判为服务器部署失败。

### 风险 1：部分文件存在编码异常

已观察到以下情况：

1. `about.html` 中存在明显乱码和部分标签异常
2. `product-detail-data.js` 中存在明显乱码内容

这意味着：

1. 即使网站成功部署，某些页面文字仍可能显示异常
2. 需要进一步做字符编码清理，通常应统一为 `UTF-8`

### 风险 2：存在中文目录与中文文件名

项目中存在：

- `素材信息/`

该目录下还有中文文件名、空格文件名、Office 文档、PDF 等。

这意味着：

1. 上传到 Linux 服务器后，要确认文件名未被转码破坏
2. Nginx 或对象存储需要正确处理 URL 编码
3. 如果外链访问异常，要优先检查文件路径和编码

### 风险 3：资源路径依赖项目根目录结构

本项目大量使用相对路径，例如：

1. `./assets/...`
2. `./site-data.js`
3. `./product-detail-data.js`
4. `./素材信息/...`

因此部署时必须保持当前目录结构，不能只上传部分文件。

### 风险 4：详情页依赖 URL 参数

`product-detail.html` 需要通过 `?code=01` 这类参数访问不同产品。

部署后应该至少测试：

1. `/product-detail.html?code=01`
2. `/product-detail.html?code=02`
3. `/product-detail.html?code=08`
